# Supabase Schema Notes

Last updated: 2026-05-07

## Scope

This document describes the first Supabase/Postgres schema pass for the VCollab incremental migration. The SQL lives in:

- `supabase/migrations/0001_baseline_schema.sql`
- `supabase/migrations/0002_storage_buckets.sql`
- `supabase/migrations/0003_initial_rls.sql`
- `supabase/seed/seed.sql`

## Conversion Strategy

- MySQL `AUTO_INCREMENT` IDs are converted to `bigint generated by default as identity`.
- Existing numeric IDs are preserved for API compatibility during migration.
- `users.auth_user_id uuid` links app users to `auth.users(id)` once Supabase Auth is enabled.
- MySQL `DATETIME(6)` becomes `timestamptz`.
- MySQL `LONGTEXT` becomes `text`.
- MySQL `JSON_SET`/string JSON becomes Postgres `jsonb`.
- MySQL conditional DDL was folded into explicit baseline columns.
- Soft-delete fields are kept: `deleted_at`, `deleted_by`.
- A shared trigger function updates `updated_at` on mutable tables.

## Auth Model

The target auth model is hybrid during migration:

- Supabase Auth owns credentials and sessions.
- `public.users` stores legacy-compatible identity, role, status, username, and email.
- `public.user_profiles` stores profile fields and counters.

This allows the old API shape to continue returning numeric IDs while new route handlers authenticate via Supabase cookies/JWTs.

## Core Tables

The baseline migration includes:

- Identity/profile: `users`, `user_profiles`
- Content: `projects`, `project_media`, `posts`, `post_media`, `blogs`, `blog_media`
- Interactions: `comments`, `comment_likes`, `likes`, `saves`, `shares`, `follows`, `follow_requests`
- Collaboration: `project_requests`, `conversations`, `conversation_members`, `messages`, `user_presence`
- Admin/moderation: `notifications`, `reports`, `warnings`, `audit_logs`, `cms_blocks`
- Targeting/search helpers: `content_targeting`, `system_tags`, `shareable_links`
- Resources: `resource_categories`, `resource_folders`, `resource_files`
- VHub: `platform_feature_settings`, `v_hub_threads`, `v_hub_replies`, `v_hub_thread_participants`

## Storage Buckets

Initial buckets:

- `profile-media`: public profile and cover images.
- `content-media`: public project/post/blog/comment media.
- `message-attachments`: private chat attachments.
- `academic-resources`: private resource files served through signed URLs or guarded endpoints.

Phase 5 should replace broad starter policies with module-specific object path rules.

## Initial RLS

`0003_initial_rls.sql` enables RLS on the first migrated module tables and adds starter policies for:

- Public profile/category/project/resource/VHub reads.
- Own profile updates.
- Project owner writes.
- Project request participant reads and owner status updates.
- Conversation member message access.
- Resource owner writes.
- Authenticated VHub writes.

Route handlers must still enforce server-side guards. RLS is the database backstop, not the only authorization layer.

## Known Follow-Ups

- Add stricter RLS for admin-only tables when admin modules migrate.
- Decide whether anonymous VHub posting remains enabled; current Spring security permits `POST /api/v1/v-hub/**`, but the starter RLS only allows authenticated inserts.
- Add RPC functions/triggers for counter updates to avoid drift.
- Add full-text or trigram search indexes once API search parity is implemented.
- Add data import scripts for existing MySQL records and local upload file migration to Supabase Storage.
